## 2. 登录与认证

5. 修改成功后会自动退出，需要用新密码重新登录

**API接口：**
```

Authorization: Basic {base64(username:oldPassword)}
Content-Type: application/json

{"newPassword": "新密码"}
```

## 3. 设备管理

### 3.1 查看设备列表

主界面左侧显示所有已注册的设备：

| 状态 | 图标 | 说明 |
|------|------|------|
| 在线 | 🟢 | 60秒内有心跳 |
| 离线 | 🔴 | 超过2分钟无心跳 |

**设备信息显示：**
- MAC地址
- IP地址
- 信号强度 (RSSI)
- 运行时间
- 固件版本

### 3.2 选择设备

1. 点击设备卡片，该设备变为选中状态（蓝色边框）
2. 选中后，右侧展开"设备工具箱"面板
3. 工具箱内的所有操作都针对当前选中的设备

### 3.3 设备心跳机制

- 设备每 60 秒发送一次心跳
- 心跳内容包含：IP地址、信号强度、运行时间、固件版本
- 超过 2 分钟无心跳，设备标记为离线

### 3.4 查看设备详情

点击设备卡片后，在工具箱的"信息查询"标签页可以查看详细信息：

| 查询类型 | 说明 |
|----------|------|
| 📋 固件信息 | 厂商、型号、固件版本 |
| 📶 信号质量 | RSRP、RSRQ |
| 💳 SIM卡信息 | IMSI、ICCID、手机号 |
| 🌍 网络状态 | 注册状态、运营商、数据连接 |
| 📡 WiFi状态 | SSID、RSSI、IP地址 |

## 4. 短信收发

### 4.1 查看接收的短信

**操作步骤：**

1. 短信历史显示在主界面下方
2. 默认显示最近 200 条短信
3. 点击"加载更多"可查看更多历史记录

**短信信息包含：**
- 接收时间
- 设备MAC
- 发送者号码
- 短信内容

### 4.2 发送短信

**操作步骤：**

1. 在设备工具箱中选择"📤 发送短信"标签页
2. 填写以下信息：
   - 目标号码：输入接收短信的手机号
   - 短信内容：输入要发送的文本
3. 点击"📨 立即发送"按钮
4. 等待设备响应（通常 5-10 秒）
5. 发送成功会显示"短信发送成功"提示

**注意事项：**
- 短信内容不宜过长（建议 70 字符以内）
- 设备必须在线才能发送
- 发送失败会显示具体错误原因

### 4.3 查看最近发送记录

发送短信页面右侧显示最近发送的 10 条记录，包括：
- 目标号码
- 短信内容
- 发送时间

## 5. 定时任务

### 5.1 创建一次性定时任务

**操作步骤：**

1. 在设备工具箱中选择"📤 发送短信"标签页
2. 填写目标号码和短信内容
3. 勾选"启用定时发送"
4. 选择发送类型："一次性发送"
5. 点击时间选择器，选择发送时间
6. 点击"⏰ 定时发送"按钮
7. 任务创建成功后会显示提示

**示例场景：**
- 定时给家人发送生日祝福
- 定时发送验证码到指定号码

### 5.2 创建周期发送任务

**操作步骤：**

1. 在设备工具箱中选择"📤 发送短信"标签页
2. 填写目标号码和短信内容
3. 勾选"启用定时发送"
4. 选择发送类型："周期发送（每N天）"
5. 选择"首次发送时间"
6. 设置发送间隔天数（如：7 表示每7天发送一次）
7. 点击"⏰ 定时发送"按钮

**示例场景：**
- 每周一早上发送周报
- 每月1号发送账单提醒
- 每天固定时间发送健康提醒

### 5.3 管理定时任务

**查看任务列表：**

1. 在设备工具箱中点击"⏰ 定时任务"标签页
2. 显示该设备的所有定时任务
3. 包括：目标号码、内容摘要、下次执行时间、状态

**查看任务详情：**

1. 点击任务卡片
2. 弹出任务详情窗口，显示：
   - 目标号码
   - 短信内容
   - 发送类型
   - 设定时间
   - 下次执行时间
   - 间隔天数（周期任务）
   - 当前状态
3. 下方显示执行历史记录

**暂停任务：**

1. 打开任务详情
2. 点击"⏸ 暂停"按钮
3. 任务状态变为"已暂停"，不再自动执行

**恢复任务：**

1. 打开被暂停的任务详情
2. 点击"▶ 恢复"按钮
3. 任务恢复执行，按原周期继续

**删除任务：**

1. 打开任务详情
2. 点击"🗑 删除"按钮
3. 确认删除后，任务被永久移除

### 5.4 执行历史

每个任务详情页面底部显示执行历史：

| 状态 | 说明 |
|------|------|
| 🟢 成功 | 短信已成功发送 |
| 🔴 失败 | 发送失败，显示原因 |

历史记录包含：
- 执行时间
- 执行状态
- 设备响应信息

### 6.4 AT终端

AT终端允许发送自定义 AT 指令直接控制模组。

**基本操作：**

1. 点击"💻 AT终端"标签页
2. 在输入框中输入 AT 指令
3. 点击"发送"按钮或按 Enter 键
4. 查看响应结果

**快捷按钮：**

| 按钮 | 指令 | 说明 |
|------|------|------|
| ATI | ATI | 显示固件信息 |
| AT+CSQ | AT+CSQ | 查询信号质量 |
| AT+CIMI | AT+CIMI | 查询 IMSI |
| AT+ICCID | AT+ICCID | 查询 ICCID |
| AT+CNUM | AT+CNUM | 查询本机号码 |
| AT+COPS? | AT+COPS? | 查询运营商 |
| AT+CEREG? | AT+CEREG? | 查询注册状态 |
| AT+CGACT? | AT+CGACT? | 查询数据连接 |

**常用 AT 指令示例：**

```bash
# 查询信号
AT+CSQ

# 查询运营商
AT+COPS?

# 激活数据连接
AT+CGACT=1,1

# 关闭数据连接
AT+CGACT=0,1

# 读取短信
AT+CMGL="ALL"
```

## 7. OTA固件升级

### 7.1 检查固件更新

**操作步骤：**

1. 在设备工具箱中选择"📦 OTA升级"标签页
2. 点击"🔍 检查更新"按钮
3. 系统自动查询服务器上的最新固件版本
4. 显示当前版本和最新版本

### 7.2 开始升级

**前提条件：**
- 设备必须在线
- 服务器已上传新版本固件

**操作步骤：**

1. 点击"🔍 检查更新"
2. 如果有新版本，"🚀 开始升级"按钮变为可点击
3. 点击"🚀 开始升级"
4. 系统自动完成：
   - 下载固件
   - 写入备用分区
   - 验证固件
   - 重启设备
5. 升级完成后设备自动重启

### 7.3 升级进度

升级过程中显示进度条，包括：
- 0% - 开始下载
- 25% - 50% - 75% - 下载进度
- 验证中 - 固件验证
- 100% - 升级完成，设备重启

### 7.4 固件管理（服务端）

**上传固件（Web界面）：**

1. 访问 Web 后台
2. 进入推送配置或其他入口
3. 找到固件上传功能
4. 选择 .bin 文件并上传
5. 填写版本号（如 1.0.2）
6. 选择平台（如 esp32c3）

**上传固件（API）：**

```bash
curl -X POST http://192.168.31.197:34567/api/ota/upload \
  -u admin:admin123 \
  -F "file=@firmware.bin" \
  -F "version=1.0.2" \
  -F "platform=esp32c3"
```

**直接放置固件：**

将固件文件放入服务器的 `firmware/` 目录：

```bash
# 文件命名格式
firmware_esp32c3_v1.0.2.bin
```

### 7.5 固件版本管理

- 每次上传固件需要版本号递增
- 系统只保留最新版本用于升级
- 旧版本固件可以保留用于回滚

## 8. 推送通知配置

2. 打开推送配置模态框

### 8.2 Telegram 推送

**配置步骤：**

1. 创建 Telegram Bot：
   - 在 Telegram 中搜索 @BotFather
   - 发送 `/newbot`
   - 设置机器人名称
   - 获取 Bot Token（如 `123456789:ABCdefGHIjklMNOpqrSTUvwxyz`）

2. 获取 Chat ID：
   - 创建机器人后，向机器人发送一条消息
   - 访问 `https://api.telegram.org/bot<TOKEN>/getUpdates`
   - 从返回的 JSON 中获取 `chat.id`

3. 在推送配置中填写：
   - 启用：☑️
   - Bot Token：`123456789:ABCdefGHIjklMNOpqrSTUvwxyz`
   - Chat ID：`123456789`

**推送格式：**
```
📱 SMS Forward

From: 13800138000
Phone: 13800138000
Time: 2026-03-20 10:30:00

验证码：123456
```

### 8.9 POST JSON 推送

**配置参数：**
- 启用：☑️
- URL：`https://your-api-endpoint.com/webhook`

推送时发送：
```json
{
  "sender": "13800138000",
  "message": "短信内容",
  "timestamp": "2026-03-20T10:30:00.000Z",
  "phone": "13800138000"
```

使用模板变量自定义推送内容。

**配置参数：**
- 启用：☑️
- URL：`https://your-api-endpoint.com/send`
- 请求体模板：
```json
{
  "title": "{sender}",
  "content": "{message}",
  "time": "{timestamp}"
}
```

**支持的变量：**
| 变量 | 说明 |
|------|------|
| `{sender}` | 发送者号码 |
| `{message}` | 短信内容 |
| `{timestamp}` | 时间戳 |
| `{phone}` | 设备SIM卡号 |

### 8.11 保存配置

## 10. MQTT命令协议

设备作为 MQTT 客户端，接收来自服务端的命令。

### 10.1 主题结构

| 主题 | 方向 | 说明 |
|------|------|------|
| `sms_forwarder/cmd/{MAC}` | 服务端 → 设备 | 发送命令 |
| `sms_forwarder/heartbeat/{MAC}` | 设备 → 服务端 | 心跳 |
| `sms_forwarder/raw_sms/{MAC}` | 设备 → 服务端 | 收到短信 |
| `sms_forwarder/resp/{MAC}` | 设备 → 服务端 | 命令响应 |

## 12. 故障排查

### 12.1 设备相关

#### 设备不在线

**可能原因：**
1. WiFi 连接断开
2. MQTT 连接断开
3. 设备死机

**排查步骤：**

1. 检查设备 WiFi 是否正常
2. 检查 MQTT Broker 是否运行
3. 重启设备（断开电源重新上电）
4. 检查固件日志输出

#### 设备收不到命令

**排查步骤：**

1. 确认设备在线
2. 确认命令格式正确
3. 检查 MQTT 主题是否匹配
4. 查看设备串口日志

#### 短信发送失败

**可能原因：**
1. SIM 卡余额不足
2. 信号质量差
3. 目标号码格式错误
4. 短信内容过长

**排查步骤：**

1. 发送 AT+CSQ 查看信号
2. 检查 SIM 卡状态 AT+CIMI
3. 缩短短信内容重试
4. 确认目标号码正确

#### Ping 测试失败

**排查步骤：**

1. 确认设备已激活数据连接
2. 检查 AT+CGACT? 状态
3. 尝试手动激活：AT+CGACT=1,1
4. 检查 APN 配置

### 12.2 服务器相关

#### Web 后台无法访问

**排查步骤：**

1. 检查 Docker 容器是否运行：`docker ps`
2. 检查端口是否冲突：`netstat -an | grep 34567`
3. 查看容器日志：`docker-compose logs -f`
4. 检查防火墙设置

#### MQTT 连接失败

**排查步骤：**

1. 检查 EMQX 是否运行
2. 检查 MQTT_HOST 配置
3. 检查网络连通性
4. 查看服务端日志

#### 定时任务不执行

**排查步骤：**

1. 确认任务状态为"执行中"
2. 确认设备在线
3. 检查服务器时间是否正确
4. 查看服务端日志中的 [Schedule] 记录

#### 推送通知不生效

**排查步骤：**

1. 确认推送通道已启用
2. 确认配置参数正确
3. 测试推送通道本身是否可用
4. 检查网络是否能访问外网

### 12.3 固件相关

#### OTA 升级失败

**可能原因：**
1. 固件文件过大
2. 固件格式错误
3. 分区配置不正确
4. 网络不稳定

**排查步骤：**

1. 检查固件大小是否超过分区容量
2. 确认固件文件完整
3. 检查 partitions.csv 配置
4. 重试 OTA 升级
5. 通过 USB 重新烧录固件

#### 固件无法启动

**排查步骤：**

1. 检查分区表配置
2. 确认固件编译选项正确
3. 查看串口日志输出
4. 重新烧录固件

### 12.4 日志查看

#### 服务端日志

```bash
docker-compose logs -f

# 查看最近日志
docker-compose logs --tail=100
```
